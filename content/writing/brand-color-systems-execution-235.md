Brand color systems fail silently. The original guidelines specify four carefully chosen colors, and three years later the brand is using seventeen. Nobody made a deliberate decision to expand the palette — it grew through accumulated improvisation. Each new application found the existing colors insufficient, and a new color was added.

By the time someone notices the drift, the system has already been compromised. Retroactively restoring discipline is much harder than building a system robust enough to prevent drift in the first place.

## Why Systems Drift

The proximate cause of color drift is almost always that the original system didn't anticipate the actual range of applications. Four colors that work for a website and a business card don't necessarily work for a data visualization, a retail environment, a trade show booth, a mobile app with multiple functional states, and a product packaging line.

When a new application arrives with needs the existing palette can't meet, the implementer has a choice: adapt creatively within the system, or add a color. Creative adaptation requires deep system understanding. Adding a color is faster.

The solution isn't stricter policing. It's designing the system to be comprehensive enough that adaptation is always possible without addition.

## What a Comprehensive System Looks Like

A brand color system capable of surviving real-world execution needs:

A primary palette with sufficient range — not just the "brand colors" but a full tonal range for each, so that lighter and darker variants are specified rather than invented.

A functional color subset — colors designated for specific functional uses like error states, success states, data visualization, or call-to-action contexts. These may overlap with the brand palette or be separately specified.

A neutral family with sufficient depth — typically five to seven values from near-white to near-black, temperature-matched to the brand palette.

Clear usage rules for each component — not just what the colors are, but when to use each one, what combinations are permitted, and what combinations to avoid.

## Building for Dark and Light Contexts

Most brands now live in both light-mode and dark-mode contexts. A color system designed only for light contexts will produce improvised dark versions that don't feel like the same brand.

I build light and dark variants into every brand color system I deliver. The dark palette isn't simply the light palette inverted — it requires rethinking contrast relationships, adjusting saturation to compensate for optical differences, and ensuring that functional colors remain clearly readable in both contexts.

## The Expansion Protocol

Even the most comprehensive system will eventually face a use case it doesn't cover. Rather than leaving this to improvisation, I include an expansion protocol in brand guidelines: a defined process for evaluating whether a new color is needed, and if so, how to derive it in a way that maintains system coherence.

This might specify: any addition must be a tonal variant of an existing palette hue; any new hue must be approved at senior brand management level; new colors are added to the guidelines before use, not after.

The protocol doesn't prevent growth — it channels it.

## Testing the System in Use

Before delivering a brand color system, I stress-test it against the actual use cases the brand faces. I mock up the colors in the specific applications — the website, the printed document, the packaging, the signage — and check whether the system covers the full range without requiring improvisation.

Gaps found during testing can be addressed in the system design. Gaps found after delivery produce drift. Do the stress test first.
